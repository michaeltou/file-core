🔷 File Gateway Core

文件网关内核
系统架构设计

一个文件解析与数据入库的中间件服务，将多种格式的外部文件解析为结构化数据， 经过滤、映射、转换后写入 OceanBase 数据库。

📅 版本 1.0 📅 2026-06-03 🐍 Python / Flask
01

系统概述

系统定位

文件网关内核（File Gateway Core）是一个文件解析与数据入库的中间件服务，负责将外部来源的多种格式文件解析为结构化数据，经过过滤、字段映射、类型转换等加工后，写入 OceanBase（Oracle 兼容模式）数据库。

核心能力

📁

多格式文件解析

支持 7 种数据源格式，每种格式有独立的解析引擎

🔧

数据加工流水线

过滤 → 映射 → 转换 → 入库，四阶段串行流水线

⚡

大文件多进程

DBF >100万行、TXT >100MB 自动切换多进程并行

📐

数据驱动配置

处理规则通过 JSON 传入，无需修改代码即可支持新业务

🔗

上下文变量体系

标量 [VAR] 和集合 [[COL.FIELD]] 两级变量替换

系统边界

外部调用方资金清算系统等
文件网关内核File Gateway Core
OceanBaseOracle 兼容数据库
文件系统DBF / TXT / Excel / CSV / XML
Redis并发控制 / 分布式锁
T2 API外部数据服务
本系统不负责：文件生成、业务校验规则定义、前端界面、数据查询服务

架构风格与设计原则

架构风格：管道-过滤器（Pipe-and-Filter）

文件读取DataFrame
→
过滤器FilterEngine
→
字段映射器FieldMapping
→
逻辑处理器ProcessEngine
→
清洗引擎CleanEngine
→
数据库写入to_sql()

每个阶段独立封装，通过 pandas DataFrame 作为管道间的标准数据载体

设计原则

📐 数据驱动

处理规则通过 JSON 配置传入，引擎根据配置动态执行，无需为不同业务编写代码

🔀 策略模式

文件类型通过 FileType 枚举分派到独立处理器，新增格式只需增加处理器

🧩 关注点分离

文件读取、数据加工、数据库写入各自独立，互不耦合

🛡️ 渐进式降级

DBF: simpledbfdm → dbfreaddm；Excel: pd.read_excel → xlsx2csv

核心领域对象 — Context（上下文）

Context 是贯穿整个处理流程的核心状态对象，采用双字典同步设计：

_context（带括号 · 变量替换）
[FUND_ID]→ "001"
[BUSINESS_DATE]→ 20241212
[FILE_PATH]→ "/data/file.dbf"
_simple_context（无括号 · SQL 绑定参数）
FUND_ID→ "001"
BUSINESS_DATE→ 20241212
FILE_PATH→ "/data/file.dbf"
大小写不敏感

所有 key 自动转大写

双字典同步

set/remove 时自动更新

集合类型支持

[[COLLECTION.FIELD]] 变量

核心领域对象 — FlowNode & InvokeMode

FlowNode 流程节点类型

1MOVE_DATA_NODE读取文件 → 过滤 → 映射 → 入库
2EXECUTE_SQL_NODE执行 DML SQL（带上下文参数绑定）
3BUILD_CONTEXT_NODE从 SQL/XML/常量构建上下文变量
4CHECK_NODE校验数据或条件，不通过则中断流程

InvokeMode 调用模式

模式值写库返回数据
NORMAL1✅❌
TEST_NO_WRITE2❌✅ 前10行
TEST_WRITE3✅✅ 前10行

核心数据流

主处理流程

HTTP POST /execute/readJSON 请求体包含文件路径、读取规则、流程节点列表
① 请求解析与上下文初始化创建 Context → 注入标准变量 [FUND_ID] [BUSINESS_DATE] [UUID] → 合并 contextMap → 限流检查 (9次/秒/进程)
② 遍历 flowNodeListmove_data_engine.move_data() → 读取 implement_type → PAGE 模式按 FileType 分派 / SCRIPT 模式 exec() 执行脚本
③ 文件读取按文件类型分派到独立处理器 (DBF/TXT/Excel/CSV/JSON/XML/T2) → 生成 pandas DataFrame
④ 数据加工流水线 (MigrateCoreEngine.dataframe_to_oracle)4a. 过滤 (FilterEngine) → 4b. 字段映射 (FieldMapping) → 4c. 入库 (to_sql) → 4d. 后置SQL (CleanEngine)
⑤ 构造响应NORMAL: {code:200} / TEST: {code:200, data:{dataList, fieldNameList}} / 异常: {code:-500} / 限流: {code:-429}

大文件处理流程

检测到大文件DBF >100万行 / TXT >100MB / 接口ID=000001
设置 [READ_BIG_FILE_FLAG] = True · 跳过限流检查
DBF 大文件记录范围分区
total_records / pool_size
TXT 大文件先分块读取到内存
pd.read_csv(chunksize)
其他格式走普通路径
multiprocessing.Pool.starmap()每个子进程：独立创建 DB 引擎 (pool=1) → 独立调用 dataframe_to_oracle() → 完成后 dispose()
Redis 并发控制 (仅 DBF)SET NX key=file_path TTL=60s · 防止同一文件并发读取 · 处理完毕后释放锁

组件详细设计

FilterEngine 过滤引擎

职责：根据 filterLogic 表达式对 DataFrame 进行行级过滤 · 执行机制：pandas DataFrame.query()

阶段1
标量变量替换：[VAR] → 从 Context 取值直接替换
例: "FUND_ID == '[FUND_ID]'" → "FUND_ID == '001'"
阶段2
集合变量替换（逐行展开）：[[COLLECTION.FIELD]] → 从 Context 取集合，逐行替换后 query，结果拼接
例: 遍历 MY_LIST 每一行，生成多个 query 表达式，合并结果

⚠️ 约束：每个过滤表达式只允许一个集合变量

FieldMapping 字段映射引擎

职责：将源字段映射到目标字段，并调用 ProcessEngine 进行类型转换和逻辑处理

输入: sourceField
[RECNUM]生成行号列 (1,2,3,...)
[UUID]生成32位hex UUID列
[UUID_10]生成十进制UUID列
其他字段: 替换 [VAR] → 得到实际列名 → 在 DataFrame 中查找 → 找到则使用该列数据 / 未找到则从 Context 取值作为常量列 → 调用 ProcessEngine

ProcessEngine 逻辑处理引擎

阶段1: 类型强制转换

STRINGstr() + strip() (除非 keep_original_space)
DECIMALpd.to_numeric(errors='coerce')
DATETIMEpd.to_datetime(format='mixed', errors='coerce')

阶段2: 业务逻辑处理

SIMPLE(1)"$" 替换为 "x"，eval("x + 2") 逐行执行
ORIGINAL(2)exec() 执行任意 Python 代码，操作整个列

数据库架构

双连接模型

🗄️ OceanBaseDbUtil（单例 · 共享连接池）
pool_size 3
max_overflow 20
pool_recycle 1800s
pool_timeout 30s
线程1 ──→ 连接池 ──→ OceanBase
线程2 ──→ 连接池 ──→ OceanBase
适用 普通文件处理路径
⚡ OceanBaseMultiProcessDbUtil（每次新建 · 每进程独立）
pool_size 1 (硬编码)
max_overflow 无
生命周期 使用后 dispose()
子进程A ──→ Engine_A (pool=1) ──→ OceanBase
子进程B ──→ Engine_B (pool=1) ──→ OceanBase
适用 大文件多进程路径

设计原因：SQLAlchemy Engine 的连接池在 fork() 后不安全，子进程必须创建独立的 Engine，不能继承父进程的连接池。

SQL 变量绑定机制

Context: [NAME] = 'Tom', [AGE] = 25
原始 SQL: SELECT * FROM T WHERE name = [NAME] AND age = [AGE]
SqlUtil.replace_sql():
1. PURE_STR_REPLACE([VAR]) → 直接内联字符串值（用于 IN 子句等）
2. 其余 [VAR] → :VAR 绑定参数语法
转换后: SELECT * FROM T WHERE name = :NAME AND age = :AGE
参数: {'NAME': 'Tom', 'AGE': 25}
SQLAlchemy text(sql).bindparams(**params) → 安全执行

写入策略

3000
行数 > 50,000
2000
行数 > 30,000
1000
其他情况
表名强制小写
避免 Oracle 双引号开销
if_exists='append'
追加写入模式
NaN 统一替换
NaN/None/'nan' → Python None

并发控制机制

🚦 限流 (ratelimit)

  • 9 次/秒/进程
  • needRateLimitControl=1 或缺省时启用
  • needRateLimitControl=2 时跳过
  • /execute/read_big_file 不限流
  • 建议值: limit.calls = threads - 1

🔒 Redis 文件锁 (仅 DBF)

  • SET NX key=file_path TTL=60s
  • 防止同一 DBF 文件被并发读取
  • 处理完毕后释放锁
  • 超时自动释放 (60s)
  • 并发读取导致严重性能退化

安全考量

风险点评估

风险严重程度说明
远程代码执行 🔴 严重 /execute/script 和 /execute/filesScript 通过 exec() 执行用户代码，无沙箱或权限控制
数据库凭证明文 🟡 中等 application.yaml 中数据库密码以明文存储
SQL 注入 🟢 低 SQL 参数通过绑定参数执行；但 PURE_STR_REPLACE 直接内联值仍有风险
进程逻辑执行 🟡 中等 ProcessEngine 的 SIMPLE(eval) / ORIGINAL(exec) 模式可执行任意 Python 代码
文件路径遍历 🟡 中等 文件路径来自请求参数，未做路径合法性校验
Redis 无认证 🟡 中等 Redis 连接未配置密码

现有防护措施

🛡️
SQL 参数绑定
防止 SQL 注入
🚦
限流机制
防止突发流量
🔒
Redis 文件锁
防止并发资源争用
♻️
连接池回收
防止连接泄漏

大文件独立部署（推荐）

🖥️ 普通文件服务 (端口 27777)
workers=1, threads=10
限流: 9次/秒
进程池: 不使用
检测到大 DBF → HTTP 转发到 28888
🖥️ 大文件服务 (端口 28888)
workers=1, threads=少量
限流: 无
进程池: 4 workers
专用于大文件处理

设计意图：大文件服务进程少但共享一个进程池（1×4=4个处理进程），避免每个 Gunicorn Worker 各自创建进程池导致资源膨胀。
